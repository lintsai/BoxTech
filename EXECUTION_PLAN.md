## 📝 開發原則

### 1. 漸進式開發

- 先求有,再求好
- 每個功能先做最小版本
- 測試通過後再優化

### 2. 持續測試

- 每個功能完成立即測試
- 保持代碼庫隨時可運行
- 發現 bug 立即修復

### 3. 文檔同步

- 代碼和文檔同步更新
- 記錄重要決策
- 記錄遇到的坑

### 4. 版本控制

- 每天至少 commit 一次
- 使用有意義的 commit message
- 重要功能完成後打 tag

### 5. 效能優先

- 先實現功能,再優化效能
- 但不要寫明顯低效的代碼
- 定期進行效能測試

## 🚨 風險管理與應變計劃

### 可能遇到的問題與解決方案

#### 問題 1: MediaPipe 準確度不足

**徵兆**: 動作識別準確率 < 80%**應變**:

- 增加訓練資料
- 調整模型架構
- 備用方案: 換用 OpenPose

#### 問題 2: GPT-4 成本過高

**徵兆**: 每月費用 > 預算**應變**:

- 增加建議模板庫,減少 API 呼叫
- 使用 GPT-3.5-turbo 替代部分場景
- 考慮訓練本地小模型

#### 問題 3: Unity 效能問題

**徵兆**: WebGL 載入時間 > 20 秒**應變**:

- 簡化 3D 模型
- 減少貼圖解析度
- 使用 LOD (Level of Detail)
- 考慮改用 Three.js

#### 問題 4: 開發進度落後

**徵兆**: 落後計劃 2 週以上**應變**:

- 重新評估優先順序
- 砍掉非核心功能
- 延後 3D 對戰到 2.0 版本
- 聚焦 MVP 核心價值
